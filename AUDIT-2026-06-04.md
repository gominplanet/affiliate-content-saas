## What shipped (6 commits + plugin v1.0.37→v1.0.39)

### Batch 1 — `7e3331e` Plugin v1.0.38: WP plugin P0/P1 batch

**Security:**
- **W01 customizations save** — deep-sanitize the array on POST (`wp_kses_post` for HTML slots, `esc_url_raw` for URLs, hex-allowlist for colors). Closes the admin-XSS path that compounds with W02 — without it, any compromised admin session ships JavaScript site-wide.
- **W02 block renderer** — `wp_kses_post` on `$block['html']` at render. Defense-in-depth in case a poisoned `affiliateos_customizations` option pre-dates W01.
- **W04 logo banner** — `esc_attr` / `esc_js` on `$bg` in inline CSS + JS contexts.
- **W08 /proxy brute-force** — was a single site-wide counter. An attacker could spam 5 bad tokens and lock out legit MVP backend calls for 60 seconds. Now per-IP counter (5/min lockout) + per-site backstop (50/min). Successful auth clears the IP counter.
- **W10 /proxy path allowlist** — biggest one. The route accepted any REST path; a leaked secret = full WP takeover via `/wp/v2/users` (create admin) or `/wp/v2/plugins` (install malicious plugin). Now hard-allowlists `/wp/v2/{posts,pages,media,tags,categories}`, `/wp/v2/users/me`, `/affiliateos/v1/*`. Plus rejects `..` segments.

**Performance:**
- **W11/P05 related-posts** — transient cache the 3-fallback query chain (12h TTL) keyed by post_id. Cuts 1-3 WP_Query hits per single-post view.
- **W14/P06 You-Might-Also-Like** — replaced `orderby=rand` (full-table sort on every page hit) with a daily-cached pool of 50 IDs + PHP `shuffle`.

### Batch 2 — `841c277` Next.js security P0s

- **S03 /api/wordpress/oauth-callback** — *critical*. Without this, an attacker who runs `/oauth-start` with `siteUrl=evil.com` + their session can trick a victim into authorizing on evil.com — the victim's freshly-minted Application Password gets persisted to the **attacker's** integrations row, AND leaks to attacker's host via our credential-test fetch. Full WP API takeover. Now requires `state.userId === current session user.id`. Session-lapsed users get bounced to re-login (small UX cost, closes account-takeover surface).
- **S02 /api/blog/product-finder** — SSRF via `body.site`. The fetch built `wpBase` from user-supplied `body.site` after a hostname match — but `http://169.254.169.254@trusted.com` parses with host=trusted.com (matches the whitelist) and the actual fetch goes to the metadata IP. Fix: build `wpBase` from the DB-trusted `match.wordpress_url` instead.
- **S04 /api/auth/callback** — open-redirect via URL-encoded slashes. `safeNext` rejected `//` and `/\` but not `/%2fevil.com` or `/%5cevil.com`. Some redirect handlers decode after the origin check and leak the post-login redirect with the session cookie. Now also rejects `%2f`/`%5c`/control chars.

### Batch 3 — `e1b847c` /preview/ admin gating + dead routes

- **D01 /preview/* was a privacy leak** — any logged-in user could see the unreleased dashboard redesign. Split the layout: server-component `layout.tsx` now redirects non-admins to `/dashboard`; the existing client chrome moved to `PreviewClientShell.tsx`. #143 (port redesign into live routes) stays blocked, but the leak is closed.
- **D03/D07/D08/D11 dead routes deleted** — 4 routes with zero callers across `app/`, `lib/`, `components/`: `/api/wordpress/connect` (replaced by connect-token + connect-and-setup), `/api/wordpress/theme` (orphan), `/api/wordpress/debug-state` (replaced by typed routes), `/api/geniuslink/test` (one-off smoke test). **~220 LOC removed.**
- **P02 buying-guides caching** — added `next: { revalidate }` to the remaining WP REST fetches (tag lookups, posts list). Cuts repeat round-trips on dashboard reload.

### Batch 4 — `033d6de` Frontend P0s

- **F01 Billing infinite spinner** — `load()` had `if (!user) return` outside any `finally` + no try/catch around the supabase chain. Either trip left the page on "Loading…" forever. Now wrapped in `try/finally` that always exits the loading state. Bonus: `monthStart` now built in UTC so users west of UTC don't see wrong counts the first/last hours of every month (F11).
- **F02 Agency no error path** — `refresh()` had `try/finally` only. A `res.json()` throw or network blip left `state=null` → user landed on the paywall card even on Pro. Added `catch` + toast.
- **F03 Customize save no error/success toast** — inline "Saved!" chip sat below 700+ lines of UI; user couldn't tell anything happened. Added `toast.success` / `toast.error`.
- **F04 Assistant silent delete + no confirm** — destructive action with `.catch(() => {})` and no confirm dialog. Wired `useConfirm` + `ConfirmHost` + surfaced errors via toast.
- **F05 Comparison tier load swallow** — Promise chain had no `.catch`; tier stayed `null` forever. Default to `'trial'` on error or no-user. Also refactored to async/await with cancel-on-unmount.

### Batch 5 — `8e6dab4` Performance P0s

- **P03 /api/seo/indexnow per-site serial** — 3-site Pro users paid 3× the wall time for credential lookup + WP `/status` fetch (10s timeout!) + IndexNow submit. Promise.all the entire per-site flow → gates on the slowest single site, not the sum. ~2s win on multi-site.
- **P09 /api/blog/generate user-image loop** — 4 user-supplied body images uploaded sequentially via `uploadImageFromUrl` (~700ms each = 2.8s avoidable). Now parallel. (The AI-generated path further down was already doing this; gap closed.)
- **P11 /api/assistant/chat insert + update serial** — independent rows; Promise.all shaves 150-300ms before the stream starts.
- **P13 /api/wordpress/wp-status two UPDATEs serial** — wordpress_sites + integrations updates now run together.
- **P04 /content fetchAllVideos `select('*')`** — included the `transcript` column (50KB+ per row, only used by the on-demand AI re-write modal). Explicit column list drops a 500-video creator's dashboard payload from **~25MB to ~3MB**.

## What didn't ship + why

### Deferred to your call — bigger refactors

- **P01 /api/seo/overview unbounded reads** *(5-50MB transfer per dashboard load)* — Loads every `blog_posts` row with the full `content` HTML for the score calculation. Real fix needs: pagination cursor, OR cache `score_detail` on `post_seo` and re-score on save. That's a half-day change touching the SEO page render path; too risky for a single overnight pass. Open as a follow-up task.
- **D02 /landing-preview/page.tsx** *(2195 LOC, public-facing)* — Currently a client-only mockup, would need the same server/client split I did for `/preview/`. Skipped because the docstring says it's intentional WIP marketing copy; the gating decision is yours.
- **S05 /api/admin/run-encryption-migration** — needs a `?confirm=<env-token>` guard + per-row dry-run-then-decrypt-check. Admin-only route, so the blast radius is bounded to compromised-admin scenarios. Defer.
- **F07 Studio / SEO / Assistant modals missing focus traps** — Three different modal sites, each requires wiring `useModalA11y`. Bundled-not-fixed because each modal has its own state shape and verifying a11y on the live UI is safer than a blind sweep.
- **F06 Agency dark mode** — 5+ hardcoded `text-gray-*` / `bg-gray-*` classes need `dark:` variants. Cosmetic, not breaking; defer to a UI polish pass.

### Reasonable to keep

- **D04 encryption migration tooling** — One-time admin pages that have ALREADY been used; could delete but keeping for auditability + the eventual key-rotation use case.
- **D05 `wordpress-theme/kadence-affiliate-child/`** *(716 LOC)* — Old theme; lives in the build but isn't shipped. Will delete next time we touch the build script.
- **D14 designer-text playground** — Admin-only tool; the team may still use it occasionally. Verified before deleting.

### Audit was wrong / not actionable

- **W03 IndexNow key endpoint headers** — `nocache_headers()` suggestion is cosmetic; the file is already served at the right content-disposition and the key is hex (no XSS risk). Not shipped.
- **S08 /api/proxy-short/[videoId]** — Cache-on-supabase-signed-URL is a theoretical risk; in practice Supabase Storage signed URLs we proxy are public-bucket URLs that don't carry credentials. Not shipped.
