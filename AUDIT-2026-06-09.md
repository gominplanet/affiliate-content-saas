## ⚠️ MUST FIX BEFORE INCREMENT C (the dormant queue cluster)

These are **P0/P1 once the queue is live** (increment C wires a producer + you run migration 119). They're interconnected — fix them together when building increment C. None can fire today.

- **[P0] Retried job misclassified as a "rewrite".** Attempt 1 creates the `blog_posts` row; any retry then sees it and hits the interactive rewrite gate (`blog/generate` ~L261-315) → 403 for non-Pro (job fails for a post that may have published), or silently burns the Pro 1-rewrite budget. **Fix:** in service mode, bypass the rewrite gate and treat an existing row as an idempotent update of the same job, not a user rewrite.
- **[P0] `done` fires while `after()` is still running ~200s.** `runBlogJob` awaits only the HTTP response (which returns before the `after()` image/fact-check/cache work). The poller reports `done` while images/SEO are still populating, and `result` never reflects them. **Fix:** in service mode, run the heavy work **inline (awaited) before returning** — gate the `after()` deferral on `!isServiceCall` (the worker has the 290s budget; there's no user request to keep fast).
- **[P0] >290s generation → abort + re-queue → concurrent double-run.** The worker's `AbortSignal` cancels its client side only; the generate invocation keeps running. The re-queued job is claimed next tick and runs a 2nd concurrent pipeline on the same video → duplicate WP posts, double image spend. **Fix:** on the worker's own client-timeout, do **not** re-queue (leave it `running`; let the stale-window recover only if it truly died). Better: have the generate route write terminal job status from inside generation (single writer).
- **[P1] `completeJob` throwing re-queues a succeeded job** (`process-generation-jobs` L63-70 — completeJob is inside the run-try, so its failure → `failJob` → re-run). **Fix:** move `completeJob` out of the run-try; a failed completion must never become a re-generation.
- **[P1] Generation quota consumed per-attempt** (`checkGenerationLimit` is a `try_consume`, not a read). A job retrying before the row exists burns up to 3 monthly units for one post. **Fix:** consume once per job (at enqueue, or idempotently keyed on job id); skip on retries.
